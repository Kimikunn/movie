# 数据接口文档

## 目标
- 本文档只服务于“任务 7：数据接入与状态完善”。
- 采用“先定义一个对象 -> 立刻一起检查 -> 再进入下一个对象”的推进方式。
- 当前以接口契约优先，先统一返回格式，再逐步落业务对象，避免前后端反复改结构。

## 当前约定
- 项目定位：个人自用，简单优先
- 接口前缀暂定：`/api`
- 业务实体统一使用 `MediaItem`，不使用 `Title`
- `contentType` 当前只支持：`'movie' | 'season'`
- 电视剧按“单季”处理，每一季作为一个独立影视条目参与搜索、收藏、观影、短评与统计
- 能不加的包装字段尽量不加
- 文档中的“待确认”字段，在你确认前都不进入最终实现

## 对象 01：统一响应对象 `ApiResponse<T>`（轻量版，已确认）

用途：
- 先只保留前端真正会用到的字段
- 不引入 `requestId`、`traceId`、`timestamp` 这一类个人项目暂时用不上的元信息
- 后续 `MediaItem`、`WatchLog`、`Stats` 都沿用这一套最小结构

```ts
type ApiResponse<T> = {
  data?: T
  error?: ApiError
}

type ApiError = {
  code?: string
  message: string
  details?: Record<string, string[] | string>
}
```

成功示例：

```json
{
  "data": {
    "id": "dune-2"
  }
}
```

失败示例：

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "参数校验失败",
    "details": {
      "mediaId": [
        "不能为空"
      ]
    },
  }
}
```

### 当前结论
- 成功返回：`{ data }`
- 失败返回：`{ error }`
- 状态码先简化为：`200/201/204/400/404/500`

## 对象 02：影视条目对象 `MediaItem`（已确认）

用途：
- 首页、搜索页、详情页、各类统计榜单都会引用它
- 先定义“影片基础信息”，不把“用户观影记录”“用户短评”“统计结果”混进来
- 第三方平台评分先不放进 `MediaItem` 主体，下一对象单独定义，避免一个对象塞太多职责

```ts
type MediaItem = {
  id: string
  contentType: 'movie' | 'season'
  title: string
  originalTitle?: string
  year: number
  runtimeMinutes: number
  genres: string[]
  countries: string[]
  directors: string[]
  cast: string[]
  synopsis: string
  posterUrl?: string
}
```

示例：

```json
{
  "id": "dune-2",
  "contentType": "movie",
  "title": "沙丘 2",
  "originalTitle": "Dune: Part Two",
  "year": 2024,
  "runtimeMinutes": 166,
  "genres": ["科幻", "冒险"],
  "countries": ["中国大陆", "美国"],
  "directors": ["丹尼斯·维伦纽瓦"],
  "cast": ["提莫西·查拉梅", "赞达亚"],
  "synopsis": "保罗接受命运，与弗雷曼并肩作战。",
  "posterUrl": "https://example.com/posters/dune-2.jpg"
}
```

为什么先拆成这样：
- 现在前端里有很多 `metaLine`、`region`、`director`、`cast` 这样的拼接字符串，但这会让后面搜索、筛选、排序都很别扭
- 如果后端直接给结构化字段，前端自己拼接展示文案，后续改页面时更稳
- `posterTone`、`tone` 这类字段是当前静态 UI 的占位写法，不应该进入正式接口

### 当前结论
- 已确认：`cast` 保留完整演员列表
- 已确认：`posterUrl` 保留，后续接真实地址
- 已确认：年份、时长、类型等基础信息采用“拆字段返回”
- 已确认：统一使用 `MediaItem`，电视剧按季处理，`contentType` 使用 `'movie' | 'season'`

## 对象 03：影视条目平台评分对象 `MediaRatings`（已确认）

用途：
- 服务详情页“我的评分对比卡”和“平台评分区”
- 这里只放第三方平台分数与聚合分，不混入用户自己的观影评分
- 用户自己的评分仍然应该从观影记录里计算，避免出现两份“我的评分”来源

```ts
type MediaRatings = {
  mediaId: string
  platformAverageScore?: number
  douban?: {
    score: number
    count?: number
    stars?: number
  }
  metacritic?: {
    score: number
    summary?: string
  }
  tmdb?: {
    score: number
  }
  imdb?: {
    score: number
    count?: number
  }
  rottenTomatoes?: {
    criticsScore?: number
    audienceScore?: number
  }
}
```

示例：

```json
{
  "mediaId": "dune-2",
  "platformAverageScore": 8.5,
  "douban": {
    "score": 8.3,
    "count": 262791,
    "stars": 4
  },
  "metacritic": {
    "score": 79,
    "summary": "Generally Favorable"
  },
  "tmdb": {
    "score": 84
  },
  "imdb": {
    "score": 8.5,
    "count": 3200000
  },
  "rottenTomatoes": {
    "criticsScore": 92,
    "audienceScore": 88
  }
}
```

为什么这样拆：
- 当前前端里有 `8.5/10`、`84%`、`262,791人评价`、`★★★★☆` 这类展示字符串
- 这些字符串更适合前端自己格式化，接口层尽量只给数字
- 这样后续你换 UI 样式、换显示单位、做国际化都不会改接口

### 当前结论
- 已确认：平台评分字段统一存数字，不带 `%`、`/10`、`人评价` 等展示文案
- 已确认：`douban.stars` 保留
- 已确认：`platformAverageScore` 由后端直接返回

## 对象 04：观影记录对象 `WatchLog`（已确认）

用途：
- 服务详情页“观影记录”列表
- 服务新增记录、修改记录
- “我的评分”优先从用户自己的观影记录里取最近一条或指定规则计算
- 后续离线缓存时，这个对象也会是重点缓存对象

```ts
type WatchLog = {
  id: string
  mediaId: string
  watchedOn: string
  watchMethod: string
  userScore: number
  note?: string
  createdAt: string
  updatedAt: string
}
```

示例：

```json
{
  "id": "log_20260308_001",
  "mediaId": "dune-2",
  "watchedOn": "2026-03-03",
  "watchMethod": "影院",
  "userScore": 9.0,
  "note": "二刷，后半段比第一次更喜欢。",
  "createdAt": "2026-03-08T12:00:00.000Z",
  "updatedAt": "2026-03-08T12:00:00.000Z"
}
```

为什么先这样设计：
- 当前已经决定去掉“观影地点”，只保留“观影方式”
- `userScore` 改成数字后，前端滑杆和输入框、统计平均分都更直接
- `createdAt` / `updatedAt` 先保留，后续做排序、同步、离线合并时更稳

### 当前结论
- 已确认：观影时间只存日期，例如 `2026-03-03`
- 已确认：去掉 `place`，只保留 `watchMethod`
- 已确认：`watchMethod` 使用字符串
- 当前前端使用的推荐值：`影院`、`流媒体`、`蓝光/碟片`、`其他`
- 已确认：`userScore` 使用数字类型

## 对象 05：短评对象 `ShortReview`（已确认）

用途：
- 服务详情页「我的短评」展示与编辑
- 当前交互是一个影视条目只有一条当前短评，点击后直接覆盖保存
- 先不引入版本历史、草稿箱、公开可见性这类复杂能力

```ts
type ShortReview = {
  id: string
  mediaId: string
  content: string
  updatedAt: string
}
```

示例：

```json
{
  "id": "review_dune-2",
  "mediaId": "dune-2",
  "content": "视听震撼，节奏流畅，第二部真正立住世界观。",
  "updatedAt": "2026-03-08T12:30:00.000Z"
}
```

为什么先这样设计：
- 当前前端本地存储其实是 `mediaId -> reviewText` 的极简映射
- 如果接口层也保持“一个影视条目一条当前短评”，前端逻辑会最简单
- 补一个 `updatedAt` 就足够支持“最后更新时间”或离线同步判断

### 当前结论
- 已确认：一个影视条目只有一条当前短评
- 已确认：保留 `id`
- 已确认：`content` 为空时允许保存空字符串

## 对象 06：通用影视卡片对象 `MediaCardItem`（已确认）

用途：
- 服务搜索结果、榜单、推荐位、最近观影等“电影卡片类”列表
- 这是对 `MediaItem` 的通用轻量投影，避免每个页面重复定义一套近似对象
- 页面特有字段在此基础上追加，不再重复维护同类字段

```ts
type MediaCardItem = {
  mediaId: string
  contentType: 'movie' | 'season'
  title: string
  originalTitle?: string
  year?: number
  countries?: string[]
  primaryGenre?: string
  runtimeMinutes?: number
  userScore?: number
  posterUrl?: string
}
```

示例：

```json
{
  "mediaId": "dune-2",
  "contentType": "movie",
  "title": "沙丘 2",
  "originalTitle": "Dune: Part Two",
  "year": 2024,
  "countries": ["美国"],
  "primaryGenre": "科幻",
  "runtimeMinutes": 166,
  "userScore": 9.2,
  "posterUrl": "https://example.com/posters/dune-2.jpg"

```

为什么先这样设计：
- 搜索页、统计榜单、推荐卡片本质上都是“电影卡片”
- 这些页面共享标题、海报、时长、评分等字段，没必要每页各造一个对象
- 把通用字段收敛成 `MediaCardItem` 后，搜索和榜单只追加自己的差异字段

### 当前结论
- 已确认：影视卡片类列表统一复用 `MediaCardItem`

## 对象 07：搜索接口响应对象 `SearchMediaResponse`（已确认）

用途：
- 服务搜索页结果列表、命中数量、空状态判断
- 搜索是核心功能，因此把列表项和列表响应拆开定义
- 先用最直接的分页模型，不上 cursor

```ts
type SearchMediaItem = MediaCardItem & {
  matchedBy: Array<'title' | 'originalTitle' | 'director' | 'cast'>
}

type SearchMediaResponse = {
  query: string
  items: SearchMediaItem[]
  suggestions?: string[]
  pagination: {
    page: number
    pageSize: number
    total: number
    totalPages: number
  }
}
```

示例：

```json
{
  "data": {
    "query": "沙丘",
    "items": [
      {
        "mediaId": "dune-2",
        "contentType": "movie",
        "title": "沙丘 2",
        "originalTitle": "Dune: Part Two",
        "year": 2024,
        "countries": ["美国"],
        "posterUrl": "https://example.com/posters/dune-2.jpg",
        "matchedBy": ["title", "originalTitle"]
      }
    ],
    "suggestions": ["沙丘", "Dune"],
    "pagination": {
      "page": 1,
      "pageSize": 20,
      "total": 1,
      "totalPages": 1
    }
  }
}
```

为什么先这样设计：
- 当前搜索页顶部会显示“搜索词”和“结果数量”，所以 `query` 和 `total` 都值得直接返回
- `items` 即使为空也应该返回空数组，前端据此进入空状态
- 个人项目里先用页码分页最省心，后面结果量真变大了再换 cursor 也不晚
- 搜索项复用 `MediaCardItem`，只额外追加 `matchedBy`

### 当前结论
- 已确认：保留 `query`
- 已确认：保留完整 `pagination`
- 已确认：保留可选字段 `suggestions?: string[]`

## 对象 08：统计概览对象 `RecordsOverview`（已确认）

用途：
- 服务 `/records` 页顶部时间范围、Hero 指标卡、快速统计双卡
- 这里只定义“概览摘要”，不把 TOP5、月趋势、类型偏好塞进同一个对象
- 年、月、周三个统计页后面都可以复用同一类结构

```ts
type RecordsOverview = {
  periodType: 'year' | 'month' | 'week'
  periodStart: string
  periodEnd: string
  watchedMoviesCount: number
  watchDurationHours: number
  watchedDaysCount: number
  averageMonthlyHours?: number
}
```

示例：

```json
{
  "periodType": "year",
  "periodStart": "2026-01-01T00:00:00.000Z",
  "periodEnd": "2026-12-31T23:59:59.999Z",
  "watchedMoviesCount": 128,
  "watchDurationHours": 286,
  "watchedDaysCount": 64,
  "averageMonthlyHours": 23.8
}
```

为什么先这样设计：
- 当前年度页 Hero 卡只需要“已看电影”和“观影时长”
- 年度页快速统计双卡现在改成“观影天数”和“月度均时”
- 这些都应该返回数字，单位展示如 `h`、`年`、`月` 由前端负责拼接
- 月页和周页仍然可以复用 `watchedMoviesCount` 显示“本月 / 本周影片”
- 时间显示标签由前端根据 `periodType + periodStart + periodEnd` 自己格式化，避免后端返回带展示语义的字符串

### 当前结论
- 已确认：时间范围由后端返回原始时间值，前端自行格式化标签
- 已确认：`watchDurationHours` 返回数字，前端拼接 `h`
- 已确认：不再返回平均评分
- 已确认：新增 `watchedDaysCount`
- 已确认：`averageMonthlyHours` 恢复，仅年度页必需，`month` / `week` 维度允许缺省
- 已确认：`watchedMoviesCount` 仍可用于月页和周页的“本月 / 本周影片”卡片

## 对象 09：统计榜单项对象 `RankedMediaItem`（已确认）

用途：
- 服务 `/records` 页年度 TOP5
- 服务 `/records/month` 页月度 TOP3
- 作为统计页里的影视榜单单项，不替代完整 `MediaItem`

```ts
type RankedMediaItem = MediaCardItem & {
  rank: number
}
```

示例：

```json
{
  "rank": 1,
  "mediaId": "dune-2",
  "title": "沙丘 2",
```

- 这里的复用关系是：`MediaCardItem` 是基础卡片对象，`RankedMediaItem` 是“榜单场景专用扩展”，并不是所有列表都应该复用 `RankedMediaItem`

- 已确认：保留 `rank`
- 已确认：榜单项使用 `primaryGenre`
- 已确认：`userScore` 返回数字

## 对象 10：趋势序列对象 `TrendSeries`（已确认）

用途：
- 服务 `/records` 页月度趋势卡
- 后续也可以复用到月页里的周节奏、其他统计趋势图
- 先定义通用时间序列结构，不把某个页面的视觉布局写死

```ts
type TrendPoint = {
  key: string
  label: string
  value: number
}

type TrendSeries = {
  periodType: 'month' | 'week' | 'day'
  unit: 'movies' | 'hours'
  points: TrendPoint[]
}
```

示例：

```json
{
  "periodType": "month",
  "unit": "hours",
  "points": [
    { "key": "2026-01", "label": "1月", "value": 56 },
    { "key": "2026-02", "label": "2月", "value": 46 },
    { "key": "2026-03", "label": "3月", "value": 72 }
  ]
}
```

为什么先这样设计：
- 当前年度页趋势图本质上只需要一组有序点：横轴标签 + 数值
- `key` 适合做稳定标识，`label` 适合前端直接展示
- `unit` 保留下来后，前端知道这组趋势是在画“观影部数”还是“观影时长”

### 当前结论
- 已确认：保留 `label`
- 已确认：`value` 返回数字
- 已确认：一个趋势接口一次只返回一组 `TrendSeries`

## 对象 11：类型偏好对象 `GenrePreferenceItem`（已确认）

用途：
- 服务 `/records` 页和 `/records/month` 页底部的类型偏好条形列表
- 只表达“这个周期里各类型看了多少部”，不混入时长、评分、平台分等别的信息
- 作为简单榜单项使用，不替代完整 `MediaItem` 或更复杂的统计对象

```ts
type GenrePreferenceItem = {
  genre: string
  watchedMoviesCount: number
}
```

示例：

```json
[
  { "genre": "剧情", "watchedMoviesCount": 32 },
  { "genre": "科幻", "watchedMoviesCount": 24 },
  { "genre": "动画", "watchedMoviesCount": 17 }
]
```

为什么先这样设计：
- 当前页面实际只用到“类型名 + 部数”
- 这组数据可以由 `WatchLog` 按周期筛选后，再关联 `MediaItem.genres` 聚合得到
- 条形宽度完全可以由前端根据列表最大值自己算，不需要后端额外返回百分比
- 个人项目先不做更细的类型拆分，比如主类型/副类型占比、按时长加权等

### 当前结论
- 已确认：字段名使用 `genre`
- 已确认：只返回 `watchedMoviesCount`，不返回 `percentage`
- 已确认：该对象直接作为数组 `GenrePreferenceItem[]` 使用
- 已确认：这组数据由 `WatchLog + MediaItem.genres` 聚合得到，不需要单独存储

## 对象 12：周详情每日记录对象 `DailyWatchRecord`（已确认）

用途：
- 服务 `/records/week` 页的“每日观影记录列表”
- 表达某一天的汇总信息，以及这一天对应的影片条目
- 这里不复用 `RankedMediaItem`，因为周页不是榜单，没有 `rank`
- 先直接复用 `MediaCardItem`，如果后面发现周页条目长期只用其中 3 到 4 个字段，再收成专门别名

```ts
type DailyWatchRecord = {
  date: string
  watchedMoviesCount: number
  watchDurationHours: number
  items: MediaCardItem[]
}
```

示例：

```json
{
  "date": "2026-02-25",
  "watchedMoviesCount": 2,
  "watchDurationHours": 1.8,
  "items": [
    {
      "mediaId": "dune-2",
      "title": "沙丘 2",
      "runtimeMinutes": 106,
      "userScore": 9.2
    },
    {
      "mediaId": "anatomy-of-a-fall",
      "title": "坠落的审判",
      "runtimeMinutes": 51,
      "userScore": 9.0
    }
  ]
}
```

为什么先这样设计：
- 当前页面头部只需要“日期 + 当日总时长”，这两个值应该直接来自接口，而不是后端拼接成文案
- 空白天也应该有结构化数据，这样前端才能稳定渲染整周 7 天
- `RankedMediaItem` 只适合榜单场景；周页条目如果复用它，反而会带入无意义的 `rank`
- 影视条目先复用 `MediaCardItem`，比新造一个和它几乎一样的 `WeekMovieItem` 更稳

### 当前结论
- 已确认：只保留 `date`，星期几由前端根据日期计算

## 接口公共约定

### 1. 状态码约定
- `200 OK`：读取成功、更新成功，并返回响应体
- `201 Created`：新增成功，并返回新建对象
- `204 No Content`：当前阶段先不使用
- `400 Bad Request`：参数错误、格式错误、校验失败
- `404 Not Found`：`mediaId`、`watchLogId` 或统计周期不存在
- `500 Internal Server Error`：服务端异常

### 2. 统一返回约定
- 成功统一返回：`{ data }`
- 失败统一返回：`{ error }`
- 列表字段统一返回空数组，不返回 `null`
- 可选单对象字段优先返回 `null` 或省略，但同一接口内保持稳定

### 3. 当前接口实现顺序
建议按前端打通优先级实现：
1. `GET /api/media/search`
2. `GET /api/media/:mediaId`
3. `PATCH /api/media/:mediaId/user-state`
4. `POST /api/media/:mediaId/watch-logs`
5. `PUT /api/media/:mediaId/watch-logs/:watchLogId`
6. `DELETE /api/media/:mediaId/watch-logs/:watchLogId`
7. `PUT /api/media/:mediaId/short-review`
8. `GET /api/home`
9. `GET /api/records/year`
10. `GET /api/records/month`
11. `GET /api/records/week`

### 4. 当前阶段不做
- 短评历史版本
- 收藏夹独立分页接口
- 多用户鉴权与权限模型
- OpenAPI 自动生成与 SDK 代码生成
